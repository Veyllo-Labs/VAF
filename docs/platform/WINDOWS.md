# VAF on Windows

This doc explains how VAF *runs* on Windows — the runtime stack, launcher, and the platform-specific behavior and fixes that only apply here. For step-by-step installation, see [../setup/WINDOWS_SETUP.md](../setup/WINDOWS_SETUP.md); this page does not repeat the install steps.

## Runtime stack on Windows

- **Python**: VAF runs from the project's own virtual environment. The installer uses a system Python from the supported range **3.10-3.13** (the CI-tested matrix); when none is found - including when only a NEWER, unsupported Python is installed (brand-new releases lack prebuilt wheels for key packages, so pip source builds fail) - it installs [uv](https://docs.astral.sh/uv/) (no admin rights) to provision a supported Python, then creates `venv\` and installs the dependencies. An existing venv built with an unsupported Python is recreated automatically. Interpreters live under `venv\Scripts\` (`python.exe` / `pythonw.exe`), not `venv/bin/`.
- **Desktop window / web-view backend**: The native window is a [pywebview](https://pywebview.flowrl.com/) wrapper (`vaf/core/desktop_window.py`) using the **EdgeChromium (WebView2)** backend on Windows — the system-native WebView, so no Chromium is bundled. The window icon comes from `webview.start(icon=...)` (Windows path), since the Qt icon hook only applies to the Linux backend. The Qt-only download/print/clipboard handlers in `desktop_window.py` are Linux-only and are simply skipped on Windows (WebView2 handles those natively).
- **Menu-bar / system tray**: [pystray](https://pystray.readthedocs.io/) drives the tray icon on all platforms; on Windows it runs in a background thread via `icon.run_detached()` while pywebview owns the main thread. Windows-specific tray notes (show the icon only after the event loop is ready, 16×16/32×32 icon sizes, overflow-area visibility) are in [SYSTEM_TRAY.md](SYSTEM_TRAY.md).
- **Docker services**: VAF keeps users/auth/setup and memory in a PostgreSQL/pgvector container plus Redis and other services. On Windows these run under **Rancher Desktop** (installed automatically by the installer, Apache-2.0, no license fee) or an existing **Docker Desktop** / WSL2 engine if present. The tray resolves `docker.exe` from Rancher's bundled CLI path (`rdctl` sibling, `%LOCALAPPDATA%\Programs\Rancher Desktop\...`) and will not restart an engine that is already running. See [../setup/DOCKER_SERVICES.md](../setup/DOCKER_SERVICES.md).
- **Launcher / entry point**: Startup runs `python -m vaf.main tray` (the tray host that also opens the desktop window and the local backend/frontend). The desktop shortcut (see below) targets `pythonw.exe -m vaf.main tray` so no console window appears. `run_vaf.bat` activates `venv\Scripts\activate.bat` and defaults to `tray` when no argument is given; the `vaf` CLI (e.g. `vaf run`, `vaf tray`) is installed as a console script into the venv.

## Installing & launching

Install via `install.bat` (a wrapper that runs `install.ps1` with `-ExecutionPolicy Bypass`); full details are in [../setup/WINDOWS_SETUP.md](../setup/WINDOWS_SETUP.md) and the OS-agnostic [../setup/INSTALLATION_GUIDE.md](../setup/INSTALLATION_GUIDE.md). Once installed, launch VAF from the **"VAF Agent"** shortcut on the Desktop or in the Start Menu (both generated by `scripts/create_app_shortcut.py`), which runs `pythonw.exe -m vaf.main tray` in the background — the tray icon appears in the notification area and the splash/desktop window opens. From a terminal, use the `vaf` CLI: `vaf tray` for the background app, or `vaf run` for the interactive TUI (append `--web` to also start the Web UI). `run_vaf.bat` double-clicked behaves like `vaf tray`.

## Platform-specific behavior & fixes

- **App shortcut is a `.lnk` with a generated `.ico`.** `create_app_shortcut.py` (`create_windows_shortcut`) writes "VAF Agent.lnk" to both the Desktop and Start Menu, preferring `pythonw.exe` (no console) and falling back to `python.exe`/`sys.executable`. The icon is built from the logo by `_build_windows_ico()` into a multi-size `vaf/media/vaf_icon_v6.ico` (256→16 px) via Pillow. Shortcut creation prefers `pywin32`'s `WScript.Shell` COM and falls back to a PowerShell COM call.
- **`pythonw.exe` has no console → stdout/stderr are patched.** When launched via the shortcut there is no console, so `sys.stdout/stderr/stdin` are `None`; `tray.py` reopens them to `os.devnull` at import so logging/uvicorn don't crash. Background subprocesses are spawned with `subprocess.CREATE_NO_WINDOW` (and `CREATE_NEW_PROCESS_GROUP` for the frontend/backend) so no stray console windows flash.
- **tzdata is required.** Windows has no system IANA time-zone database, so `tzdata` (a `requirements.txt` / base dependency) supplies the zoneinfo data used by `vaf/core/user_time.py` for tz-aware scheduling and quiet hours.
- **Process stop uses `taskkill`, not `os.killpg`.** POSIX process-group signaling (`os.killpg`, `start_new_session`) is not available on Windows. VAF stops the frontend/backend and tracked sub-agents with `taskkill /F /T /PID <pid>` (tree kill), or via `psutil` when present. The frontend also assigns children to a Windows **Job Object** so the whole tree is reaped.
- **OS integration paths.** Config/data/cache resolve to `%APPDATA%\vaf`, `%LOCALAPPDATA%\vaf`, and `%LOCALAPPDATA%\vaf\cache`; tray login-autostart writes a `VAF Tray.cmd` into the user's Startup folder; URLs open via `os.startfile` / browser-specific `--inprivate` (Edge) flags (`vaf/core/platform.py`).
- **Local network hosting configures the Windows Firewall** to allow only private (RFC 1918) ranges when LAN hosting is enabled — see [../setup/WINDOWS_SETUP.md](../setup/WINDOWS_SETUP.md).
- **Splash screen** shows on launch until the frontend is actually serving (avoids the `:3000` flash) — same behavior on every OS; see [STARTUP_SPLASH.md](STARTUP_SPLASH.md).

## Known limitations / gotchas

- **A container runtime is required.** Sign-in and setup live in the pgvector container, so VAF cannot complete first-run without Rancher Desktop / Docker Desktop running. On a first Rancher install its WSL2 provisioning can take a while; WSL2 itself (incl. the "Virtual Machine Platform" feature) is checked and enabled by the installer up front - one UAC prompt, usually followed by a Windows restart and re-running `install.bat`. VAF waits for the database and self-heals: the login page shows "Starting the database..." until PostgreSQL accepts connections, then the first-run setup wizard appears automatically - no restart needed.
- **`pyttsx3` was removed on Windows** because the SAPI/comtypes path caused a 1–4 GB RAM explosion; text-to-speech runs through the Docker (Piper) engine instead.
- **Tray icon may hide in the overflow area** on Windows 10/11 — click the `^` arrow in the notification area to pin it.
- **macOS/Linux-only deps are not installed**: `pyobjc-framework-Cocoa` (darwin) and `PySide6`/`qtpy` (linux) are platform-gated in `requirements.txt` / `setup.py`, so the Qt WebEngine anti-leak tuning and the Linux/Qt download-print bridges do not apply here (WebView2 covers those cases natively).
- **Port conflicts (3000/8001):** VAF tries to clear stale processes; if a port stays held, free it manually (`netstat -ano | findstr :3000` then `taskkill /F /PID <pid> /T`) — see [../setup/WINDOWS_SETUP.md](../setup/WINDOWS_SETUP.md).
