# Web UI Internationalization (i18n)

The VAF Web UI is localized using **next-intl**. All user-visible text should go through translation keys so the interface can be shown in multiple languages.

## Scope

- **In scope:** All visible text in the Web UI (Next.js app under `web/`): layout, main page, settings, connections, modals, toasts, tooltips.
- **Out of scope:** Backend and CLI messages are not translated by this system.

## Architecture

- **Library:** [next-intl](https://next-intl.dev/) (Next.js App Router).
- **Locale source of truth:** `web/lib/languages.ts` — list of `{ code, name, flag }`. Used by the language switcher and optionally by the generator script.
- **Messages:** One JSON file per locale under `web/messages/` (e.g. `de.json`, `en.json`). Master/reference locale is German (`de.json`).
- **Persistence:** The selected UI language is stored in `localStorage` under the key `ui_locale`. No backend sync.

## File Layout

```
web/
├── lib/
│   ├── languages.ts      # Supported locales (code, name, flag)
│   └── localeStore.ts    # Zustand store: locale, init(), setLocale()
├── messages/
│   ├── de.json           # German (master)
│   └── en.json           # English
├── components/
│   └── IntlProviderWrapper.tsx  # Wraps app with NextIntlClientProvider
└── scripts/
    ├── generate-locales.js      # Create new locale files from master
    └── locale-codes.json        # Optional list for script (e.g. ["en"])
```

## Adding a New Language

1. **Add the locale to the source of truth**  
   Edit `web/lib/languages.ts`: add `{ code: "xx", name: "…", flag: "…" }` to the `languages` array.

2. **Create the message file**  
   From the `web/` directory run:
   ```bash
   node scripts/generate-locales.js xx
   ```
   This copies `messages/de.json` to `messages/xx.json`. Translate the values in `messages/xx.json`.

3. **Wire the locale into the app**  
   In `web/components/IntlProviderWrapper.tsx`: add `import xxMessages from "@/messages/xx.json"` and add `xx: xxMessages` to the message map. No change is required in `web/lib/localeStore.ts`; it validates codes via `isSupportedLocale()` from `languages.ts`.

4. **Optional:** Add the code to `web/scripts/locale-codes.json` if you use the script without arguments.

## Adding or Changing Translation Keys

- **Define keys in the master:** Edit `web/messages/de.json`. Use a clear structure (e.g. `layout`, `common`, `status`, `nav`, `main`, `auth`, `settings`, `connections`, `notifications`).
- **Use in components:**  
  - `const t = useTranslations('namespace');` then `t('key')`.  
  - For interpolation: `t('key', { count: n })` and in JSON use `"key": "RAG: {count} Treffer"`.
- **Update all locales:** After changing `de.json`, update `en.json` (and any other locale) with the same keys so no key is missing.

## Language Switcher

The UI language can be changed under **Settings → Interface → Language**. The dropdown is built from `web/lib/languages.ts`. The choice is persisted in `localStorage` and applied immediately.

## Generator Script

- **With arguments:** `node scripts/generate-locales.js en fr` — creates or overwrites `messages/en.json` and `messages/fr.json` from `de.json`. Use `--force` to overwrite existing files.
- **Without arguments:** If `scripts/locale-codes.json` exists and contains an array of codes, the script creates missing locale files (skips existing ones unless `--force`).

## Best Practices

- Use namespaces in `useTranslations('namespace')` to match the structure in JSON (e.g. `settings`, `main`).
- For new UI strings, add the key to `de.json` (and other locales) and use `t('key')` in the component; do not add hardcoded user-facing text.
- Keep the master locale (`de.json`) complete; other files can fall back to missing keys at runtime if next-intl is configured with a fallback locale.
