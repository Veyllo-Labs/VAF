# Translation System — Technical Specification

This document describes the design, behaviour, and maintenance of the Web UI translation system. It is written for maintainers and contributors who need to add languages, add or change strings, or understand how the system works.

---

## 1. Purpose and scope

**Purpose.** The Web UI (Next.js app under `web/`) must display all user-visible text in the language chosen by the user. The translation system provides a single way to store and resolve that text per locale.

**Scope.**

- **In scope:** Every string that the user sees in the browser: labels, buttons, placeholders, titles, tooltips, status text, error messages, and section headings. All of these are resolved from locale-specific message files via keys.
- **Out of scope:** Text produced by the backend (Python), CLI output, log messages, and content generated by the agent (e.g. chat replies). Those are not part of this system.

---

## 2. Architecture overview

**Runtime flow.**

1. On first load, a client-side store reads the UI locale from `localStorage` (key: `ui_locale`). If none is stored or the value is invalid, the locale is derived from the browser language (e.g. `navigator.language`), then normalised to a supported code (e.g. `de`, `en`). If the browser language is not supported, a fixed default locale is used.
2. The root layout renders a client-only wrapper that subscribes to this store, loads the message set for the current locale, and wraps the app with the next-intl provider. The same wrapper sets `document.documentElement.lang` so the page language is correct for accessibility and hyphenation.
3. Any component that needs a translated string calls `useTranslations(namespace)` and then `t('key')` (and optionally `t('key', { param: value })` for interpolation). The provider supplies the message map for the active locale; next-intl resolves the key and returns the string.

**Data flow.**

- **Locale list:** Defined once in `web/lib/languages.ts` (array of `{ code, name, flag }`). Used by the language switcher UI and by the store to validate stored or browser-derived codes. Adding a new language requires an entry here and a corresponding message file and provider mapping (see below).
- **Messages:** One JSON file per locale under `web/messages/`, e.g. `de.json`, `en.json`. Each file has the same key structure; only the values differ. The German file is the reference (master): new keys are added there first, then copied or translated into other locale files.
- **Persistence:** Only the chosen locale code is persisted, in `localStorage` under `ui_locale`. No server round-trip is used. The choice applies to the current browser only.

**Libraries.**

- **next-intl:** Used for message lookup, namespaces, and interpolation. The app uses the client provider only; locale is not taken from the URL. All message data is passed in at runtime from the wrapper component.

## 3. File structure and responsibilities

| Path | Responsibility |
|------|----------------|
| `web/lib/languages.ts` | Defines the list of supported locales (`languages`), their codes, display names, and flags. Exposes `localeCodes`, `defaultLocale`, `isSupportedLocale()`, and `resolveBrowserLocale()`. No message content. |
| `web/lib/localeStore.ts` | Zustand store holding the current `locale`. On mount, `init()` runs once: it reads `ui_locale` from `localStorage`, validates it with `isSupportedLocale()`, and falls back to `resolveBrowserLocale()` or `defaultLocale`. `setLocale(code)` updates the store and writes back to `localStorage`. |
| `web/components/IntlProviderWrapper.tsx` | Client component that subscribes to the locale store, loads the message object for that locale from a static map, and renders `NextIntlClientProvider` with `locale` and `messages`. Also sets `document.documentElement.lang` when the locale changes. |
| `web/messages/*.json` | One file per locale. Flat or nested key structure; next-intl resolves dotted keys (e.g. `settings.general`). Keys must be kept in sync across all locale files. |
| `web/scripts/generate-locales.js` | Node script that copies `web/messages/de.json` to one or more other locale files (e.g. `node scripts/generate-locales.js fr`). Used when adding a new language to create an initial file with the same keys as the master; values are then translated or left as placeholders. |

**Where the locale is not stored.** The backend does not store or return the UI language. Session and user identity are separate from this; the translation system is entirely client-side and storage is local.

---

## 4. Key design decisions

**Single source of truth for the locale list.** The array in `languages.ts` is the only place that defines which locales exist. The language switcher and the store’s validation both use this. Adding a locale means: add an entry there, add a message file, and register that file in `IntlProviderWrapper`’s message map. The store already accepts any code for which `isSupportedLocale()` is true, so no change is needed in `localeStore.ts` when adding a language.

**Client-only locale resolution.** The locale is not encoded in the URL and not read from the server. This avoids middleware and server-side locale handling. The first paint may use the default locale until the client runs and reads `localStorage` or the browser language; the UI then updates to the user’s choice. This is an accepted trade-off for simplicity.

**One message file per locale.** All keys live in each file; there is no lazy loading of message files by namespace. The number of locales is small and the JSON size is modest, so a single map per locale is sufficient.

**Master locale (German).** New keys are added first to `de.json`. Other locale files are updated to include the same keys so that no key is missing at runtime. Missing keys in a non-master locale can be left as copies of the master for later translation.

## 5. Adding a new language

1. **Extend the locale list.** In `web/lib/languages.ts`, add an object to the `languages` array: `{ code: "xx", name: "Language name", flag: "🇺🇺" }`. The type `LocaleCode` and `localeCodes` will include the new code automatically.

2. **Create the message file.** From the project root, run from `web/`:  
   `node scripts/generate-locales.js xx`  
   This creates `web/messages/xx.json` with the same keys as `de.json`. Translate the values as needed.

3. **Register the messages in the app.** In `web/components/IntlProviderWrapper.tsx`, add:
   - `import xxMessages from "@/messages/xx.json";`
   - In the object passed to `NextIntlClientProvider`, add the new locale to the message map (e.g. `xx: xxMessages`). The type `LocaleCode` already includes the new code if step 1 is done.

No change is required in `localeStore.ts`: it uses `isSupportedLocale()`, which is driven by `languages.ts`.

## 6. Adding or changing translation keys

**Adding a key.**

1. Edit `web/messages/de.json`. Insert the key in the appropriate namespace (e.g. under `main`, `settings`, `common`). Use a short, stable key name (e.g. `newFeatureLabel`, not a full sentence).
2. Add the same key to every other locale file under `web/messages/` (e.g. `en.json`). Value can be the same as the master initially, then translated.
3. In the component, call `useTranslations('namespace')` (where `namespace` matches the top-level key in the JSON) and then `t('newFeatureLabel')`.

**Interpolation.** If the string contains a variable (e.g. count, name), use a placeholder in the JSON, e.g. `"ragHits": "RAG: {count} Treffer"`. In code, call `t('ragHits', { count: ragResults.sources.length })`. next-intl replaces `{count}` with the value.

**Changing or removing a key.** Update all locale files so that no file references a key that no longer exists. Remove or replace usages in components. Avoid reusing the same key for a different meaning; prefer a new key so that existing translations are not misinterpreted.

---

## 7. Using translations in components

**Basic usage.**

- In a client component, call `const t = useTranslations('namespace');` once. Then use `t('key')` for static strings and `t('key', { param: value })` when the message contains placeholders.
- The namespace must match a top-level key in the JSON (e.g. `main`, `settings`, `nav`). The key is the path under that namespace (e.g. `howCanIHelp` under `main`).

**Where hooks are allowed.** `useTranslations` is a React hook. It must be called in a component that is rendered inside `IntlProviderWrapper`. Server components do not have access to the provider’s context; for server-rendered text that should be translated, the text would need to be rendered by a client component that uses the hook.

**Fallback.** If a key is missing in the current locale’s message file, next-intl may show the key itself or a fallback depending on configuration. To avoid that, keep all locale files in sync with the same set of keys.

## 8. Best practices

**Key and namespace design.**

- Use a small set of top-level namespaces that match areas of the UI (e.g. `nav`, `main`, `settings`, `common`, `auth`). Avoid a single flat list of hundreds of keys.
- Name keys by meaning, not by the exact wording (e.g. `save`, `cancel`, `connectionStatus`). Wording may change; the key should stay stable.
- Do not put sentence-long text as key names; use short identifiers (e.g. `startConversationOrWorkflow`, not `startAConversationOrChooseAWorkflow`).

**Consistency.**

- After adding or renaming a key in `de.json`, update every other locale file in the same commit or pull request so that no locale is missing the key.
- When removing a key, remove it from all locale files and from all component usages together.

**Content.**

- Do not embed HTML or component logic in message strings. Use plain text and placeholders. If you need structured content, use multiple keys or build the structure in the component.
- Keep strings suitable for translation: avoid concatenation of multiple keys in code to form a sentence, because word order differs across languages. Prefer one key per full phrase or use interpolation.

**Testing.**

- After adding a language, switch the UI to that language in Settings and walk through the main screens to confirm that no key is missing and that interpolated values (e.g. counts) appear correctly.
- If you add a key used in a component that is only rendered under certain conditions, ensure that the key exists in all locale files even if that component is not visible by default.

## 9. Language switcher

The user changes the UI language in **Settings → Interface**. The first section in that tab is “Language” (or the equivalent in the current locale). The dropdown is built from the `languages` array in `languages.ts`; each option shows the flag and the name. On change, the store’s `setLocale(code)` is called, which updates the Zustand state and `localStorage`. The wrapper component re-renders with the new locale and message set, so the rest of the UI updates without a full reload.

## 10. Generator script

**Location:** `web/scripts/generate-locales.js`

**Behaviour.**

- **With arguments:** `node scripts/generate-locales.js xx [yy ...]` creates `web/messages/xx.json` (and optionally `yy.json`, etc.) by copying the content of `web/messages/de.json`. If a file already exists, it is left unchanged unless `--force` is passed.
- **Without arguments:** The script looks for `web/scripts/locale-codes.json`. If it exists and contains a JSON array of locale codes, it creates missing message files for those codes (again, skipping existing ones unless `--force`).

**When to use.** Use the script when introducing a new locale: run it once with the new code to create the initial JSON file, then translate or adjust the values. Do not use it to overwrite existing locale files (e.g. `en.json`) unless you intend to replace their content with the master; normally you would only run it for new codes.
